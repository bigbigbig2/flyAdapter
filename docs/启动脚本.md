# GR3 现场启动脚本

本文档只保留当前工程采用的启动方式：所有 HumanoidNav、load_map、RViz、GR3 Adapter 都统一使用命名空间：

```bash
GR301AA0025
```

ROS2 命令里一般写成：

```bash
/GR301AA0025
```

Adapter 环境变量里写成带不带 `/` 都可以，推荐：

```bash
export ROBOT_NAMESPACE=GR301AA0025
```

代码内部会自动归一化成 `/GR301AA0025`。

## 0. 总启动顺序

正确顺序是：

```plain
1. 启动 AuroraCore
2. 启动 HumanoidNav
3. 检查 ROS2 topic/service/action
4. 启动 GR3 Adapter
5. 加载地图并进入 localization
6. 检查定位状态
7. 再测试背包接口 / Web 调试页 / Swagger
```

如果第 2 步 HumanoidNav 没起来，后面的 `/slam/relocation`、`/slam/navigate_to`、`/robot/map/load` 都不会真正成功。

## 1. 启动 AuroraCore

终端一：

```bash
sudo docker start fourier_aurora_server
sudo docker exec -it fourier_aurora_server bash

cd /workspace
sed -i 's/RobotName:.*/RobotName: gr3v233/g' config/config.yaml
sed -i 's/RunType:.*/RunType: 1/g' config/config.yaml

AuroraCore --config config/config.yaml
```

这个终端保持运行，不要关闭。

## 2. 启动 HumanoidNav

终端二：

```bash
ssh -X gr301ab0113@192.168.137.220

cd /opt/fftai/humanoidnav
source /opt/ros/humble/setup.bash
source /opt/fftai/humanoidnav/install/setup.bash

./scripts/run_real.sh \
  --unified-init mapping \
  --sensor-type airy \
  --disable-ddscmd \
  --no-rviz \
  --namespace GR301AA0025
```

注意：

- 不要再混用 `GR301AB0025`。
- 不要一部分命令使用 namespace，另一部分命令不使用 namespace。
- HumanoidNav、Adapter、load_map、RViz、ROS2 检查命令都必须保持同一个 namespace。

## 3. 检查 HumanoidNav

终端三：

```bash
source /opt/ros/humble/setup.bash
source /opt/fftai/humanoidnav/install/setup.bash
export NS=/GR301AA0025

ros2 service list | egrep "$NS/(slam/set_mode|slam/load_map|slam/save_map|cancel_current_action|get_current_action)"
ros2 action list | grep "$NS/navigate_to_pose"
ros2 topic list | egrep "$NS/(robot_pose|odom|odom_status_code|odom_status_score|action_status|Humanoid_nav/health|Humanoid_nav/events|slam/mode_status|map|scan)"
```

至少要看到这些关键接口：

```plain
/GR301AA0025/slam/set_mode
/GR301AA0025/slam/load_map
/GR301AA0025/slam/save_map
/GR301AA0025/navigate_to_pose
/GR301AA0025/robot_pose
/GR301AA0025/odom_status_code
/GR301AA0025/Humanoid_nav/health
```

如果查不到，优先排查 HumanoidNav 是否启动失败、namespace 是否写错，而不是排查 FastAPI。

## 4. 启动 GR3 Adapter

终端四：

```bash
cd ~/aurora_ws/gr3

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

source /opt/ros/humble/setup.bash
source /opt/fftai/humanoidnav/install/setup.bash
python -c "import numpy; import rclpy; print('PY_ROS_IMPORT_OK')"

export ROBOT_NAMESPACE=GR301AA0025
export ADAPTER_HOST=0.0.0.0
export ADAPTER_PORT=8080
export AURORA_BACKEND=docker
export AURORA_CONTAINER_NAME=fourier_aurora_server
export AURORA_DOMAIN_ID=123
export AURORA_CLIENT_MODULE=fourier_aurora_client

chmod +x scripts/run_adapter.sh
./scripts/run_adapter.sh
```

启动后先测：

```bash
curl http://127.0.0.1:8080/healthz
curl http://127.0.0.1:8080/slam/status
curl http://127.0.0.1:8080/robot/status
curl http://127.0.0.1:8080/robot/readiness
```

判断方式：

- `/healthz` 通，说明 HTTP 进程活着。
- `/slam/status` 通，说明 Unitree 兼容接口活着。
- `/robot/status` 里 `ros.ready=true`，说明 Adapter 已经连上 ROS2/HumanoidNav。
- `/robot/readiness` 用来判断当前是否满足导航条件。

如果 `/robot/status` 里出现：

```plain
ROS2 Python imports unavailable: No module named 'numpy'
```

说明 adapter 的虚拟环境缺少 ROS2 Python 消息依赖。执行：

```bash
cd ~/aurora_ws/gr3
source .venv/bin/activate
pip install -r requirements.txt
source /opt/ros/humble/setup.bash
source /opt/fftai/humanoidnav/install/setup.bash
python -c "import numpy; import rclpy; print('PY_ROS_IMPORT_OK')"
```

然后停掉旧 adapter，重新 `./scripts/run_adapter.sh`。

如果 `/robot/aurora/state` 不通，先按文档里已经记录的容器执行思路检查：

```bash
docker ps | grep fourier_aurora_server
docker exec -w /workspace fourier_aurora_server python3 -c "import importlib.util; print(importlib.util.find_spec('fourier_aurora_client')); print(importlib.util.find_spec('aurora_sdk')); print(importlib.util.find_spec('fftai_aurora_sdk'))"
```

如果 docker 权限不够：

```bash
sudo usermod -aG docker gr301ab0113
newgrp docker
```

## 5. 加载地图进入定位模式

推荐先用 HumanoidNav 官方脚本验证底层链路：

```bash
cd /opt/fftai/humanoidnav
source /opt/ros/humble/setup.bash
source /opt/fftai/humanoidnav/install/setup.bash

./scripts/load_map.sh \
  --map-path /opt/fftai/nav/map \
  --namespace GR301AA0025
```

看到类似输出：

```plain
Successfully loaded map and switched to localization mode
```

再检查：

```bash
timeout 5s ros2 topic echo /GR301AA0025/slam/mode_status --once
timeout 5s ros2 topic echo /GR301AA0025/robot_pose --once
timeout 5s ros2 topic echo /GR301AA0025/odom_status_code --once
timeout 5s ros2 topic echo /GR301AA0025/odom_status_score --once
```

`odom_status_code` 常见含义：

```plain
0 = IDLE
1 = INITIALIZING
2 = GOOD
3 = FOLLOWING_DR
4 = FAIL
```

只有 `2 = GOOD` 才适合开始导航。

## 6. 通过 Adapter 加载地图

底层脚本验证通过后，再测 Adapter 接口：

```bash
curl -X POST http://127.0.0.1:8080/slam/relocation \
  -H "Content-Type: application/json" \
  -d '{"map_path":"/opt/fftai/nav/map","x":0,"y":0,"z":0,"yaw":0,"wait_for_localization":false}'
```

或：

```bash
curl -X POST http://127.0.0.1:8080/robot/map/load \
  -H "Content-Type: application/json" \
  -d '{"map_path":"/opt/fftai/nav/map","x":0,"y":0,"yaw":0}'
```

## 7. 启动 RViz

如果要看 RViz，使用带 namespace 的配置：

```bash
source /opt/ros/humble/setup.bash
source /opt/fftai/humanoidnav/install/setup.bash

rviz2 -d ~/aurora_ws/navigation_view_GR301AA0025.rviz \
  --ros-args \
  -r tf:=/GR301AA0025/tf \
  -r tf_static:=/GR301AA0025/tf_static
```

RViz 的 Fixed Frame 使用 `map`，不是 `/GR301AA0025/map`。namespace 是 topic 前缀，TF frame 名一般仍然是 `map`、`odom`、`base_link`、`lidar_link`。

这里保留现场已经跑通过的写法 `tf:=...`。如果 RViz 仍然报 `target_frame map does not exist`，优先排查定位链路是否已经发布 `map -> odom`，不要先改 RViz 启动命令。

如果 RViz 提示 `target_frame map does not exist`，先确认 TF 是否有数据：

```bash
timeout 5s ros2 topic echo /GR301AA0025/tf --once
timeout 5s ros2 topic echo /GR301AA0025/tf_static --once

ros2 run tf2_ros tf2_echo map odom \
  --ros-args \
  -r tf:=/GR301AA0025/tf \
  -r tf_static:=/GR301AA0025/tf_static
```

如果 `map -> odom` 不存在，并且 `/GR301AA0025/odom_status_code` 还是 `data: 1`，说明定位还在初始化，需要先加载地图、设置初始位姿或等待定位变成 `2 = GOOD`。

## 9. 最小冒烟流程

现场最小验证顺序：

```bash
curl http://127.0.0.1:8080/healthz
curl http://127.0.0.1:8080/slam/status
curl http://127.0.0.1:8080/robot/status
curl http://127.0.0.1:8080/robot/readiness

ros2 service list | egrep "/GR301AA0025/(slam/load_map|slam/save_map|cancel_current_action|get_current_action)"
ros2 action list | grep /GR301AA0025/navigate_to_pose

timeout 5s ros2 topic echo /GR301AA0025/robot_pose --once
timeout 5s ros2 topic echo /GR301AA0025/odom_status_code --once
```

这套顺序可以快速判断问题在哪一层：

- HTTP 不通：看 Adapter 进程。
- `/robot/status` 里 ROS 不 ready：看 ROS2/HumanoidNav 环境和消息包。
- ROS2 service/action/topic 查不到：看 HumanoidNav 启动和 namespace。
- `odom_status_code != 2`：看地图、初始位姿、定位质量。
- Aurora 不通：看 AuroraCore、SDK、docker 权限。
