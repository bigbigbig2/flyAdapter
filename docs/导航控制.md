**Aurora / Aurora SDK** 是底层运动控制层，负责让机器人“站住、走起来、收发机器人状态”。官方总览里明确说，GRX 系列的运动控制功能基于名为 **Aurora** 的中间层软件，Aurora SDK 则通过 **DDS** 提供 Python 接口来获取状态和下发运动命令。

**HumanoidNav** 是上层导航系统，基于 **ROS 2 Humble**，核心由 **SLAM、Nav2、Octomap、Perception、fourier_msgs** 组成：SLAM 负责建图和定位，Nav2 负责路径规划与控制，Octomap 负责 3D 到 2D 地图管理，Perception 做感知预处理，`fourier_msgs` 提供导航动作状态和监控接口。它的输入是 `/imu`、`/lidar_points` 等传感器数据，输出是地图、位姿、路径和速度命令

架构：  


```bash
GR3 真机
├── Aurora / AuroraCore
│   ├── 真机底层控制
│   ├── 姿态 / 步态 / 速度执行
│   └── DDS 通信
│
└── HumanoidNav / ROS2
    ├── 建图
    ├── 定位
    ├── 路径规划
    ├── 导航任务
    ├── 打点 / POI
    └── 巡航任务编排
```



+ `slam`：建图 / 定位模式切换 
+ `map`：地图发布 
+ `navigate_to_pose`：导航动作 
+ `/robot_pose`：机器人当前位姿 
+ `/odom_status_code`：定位状态

## 五、接口封装
可以。先把这套接口怎么“分层封装”说清楚，再逐个讲“它包了什么、什么时候用、怎么用”。

这套工程本质上是 **HTTP 适配层**。  
从实现上看，大概是这四层：

+ **FastAPI 接口层**：你看到的这些 `/robot/...` 路由
+ **Service 业务层**：把“加载地图后等待定位”“按 POI 导航”“顺序跑 patrol/mission”这种流程编排起来
+ **Bridge 桥接层**：
    - 一路走 **ROS2 CLI / HumanoidNav**，对应地图、定位、导航、动作监控
    - 一路走 **Aurora SDK**，对应站立、FSM、停运动
+ **本地 YAML 数据层**：POI、route、runtime 这些业务数据保存在本地 YAML，不是底层地图服务原生资产

所以你可以把它理解成：

**底层 ROS2/Aurora 很分散，这个工程把它们整理成一套“业务 HTTP API”。**

---

+ `GET /robot/patrol/status`

+ `POST /robot/navigation/cancel`
+ `POST /robot/aurora/fsm`
+ `POST /robot/aurora/ensure_stand`

#### 3）业务资产类

### 1.健康 / 总状态 / readiness
#### `GET /healthz`
**作用**：服务进程活没活。  
**封装了什么**：只是 FastAPI 自身健康检查，不代表机器人 ready。  
**什么时候用**：部署后先看服务是否启动。  
**怎么用**：

```bash
curl http://127.0.0.1:8080/healthz
```

---

#### `GET /robot/status`
**作用**：一次性看总状态。  
**封装了什么**：把当前地图、位姿、定位状态、健康状态、事件、动作状态、Aurora 状态、当前导航任务、当前 patrol/mission 任务聚合到一个响应里。  
**什么时候用**：调试时最常用，先看全局。  
**重点字段**：

+ `current_map`
+ `pose`
+ `odom_status_code`
+ `odom_status_score`
+ `slam_mode_status`
+ `localization_status`
+ `health`
+ `action_status`
+ `aurora_state`

底层这些状态主要来自 `/robot_pose`、`/odom_status_code`、`/odom_status_score`、`/slam/mode_status`、`/Humanoid_nav/health`、`/action_status` 等 ROS2 话题。`/robot_pose` 是 map 坐标系下的 6DOF 位姿；`/odom_status_code` 表示定位状态，`2=GOOD`；`/odom_status_score` 表示定位质量分数。

---

#### `GET /robot/readiness`
#### `POST /robot/navigation/precheck`
这两个你可以当成一类。

**作用**：判断“现在能不能安全开导航”。  
**封装了什么**：

+ 导航 action 是否存在
+ 地图/定位 service/topic 是否存在
+ 当前是否有地图
+ 当前是不是 localization 模式
+ `robot_pose` 是否新鲜
+ `odom_status_code` 是否新鲜且是 GOOD
+ `health` 有没有 error/fatal
+ Aurora 是否连通，能不能判断是否站立

底层判断依据对应 HumanoidNav 的 `/slam/set_mode`、`/slam/load_map`、`/robot_pose`、`/odom_status_code`、`/odom_status_score`、`/Humanoid_nav/health`、`/navigate_to_pose` 这些接口。

**区别**：

+ `GET /robot/readiness`：更像“状态页”
+ `POST /robot/navigation/precheck`：更像“出发前预检”

**什么时候用**：

+ `goto_pose / goto_poi / patrol / mission` 前先看一次

---

#### `GET /robot/localization/status`
**作用**：只盯定位这条链。  
**封装了什么**：

+ 当前地图
+ `slam_mode`
+ `pose`
+ `odom_status_code`
+ `odom_status_score`
+ `health`
+ `pose_fresh / odom_fresh / slam_mode_fresh`
+ blockers / warnings

**什么时候用**：

+ 你地图 load 完了，但机器人还不能导航
+ 想知道卡在“没进 localization”还是“odom 还没 GOOD”

底层语义来自：

+ `/slam/mode_status`：当前是 `mapping` 还是 `localization`
+ `/robot_pose`：定位位姿
+ `/odom_status_code`：IDLE / INITIALIZING / GOOD / FOLLOWING_DR / FAIL
+ `/odom_status_score`：定位评分

---

### 3.SLAM / 地图 / 定位
#### `POST /robot/slam/mode`
**作用**：切换建图 / 定位模式。  
**封装了什么**：底层 ROS2 service `/slam/set_mode`。  
`mode` 支持 `mapping` 或 `localization`。`SetMode` 文档里也说明了，从建图切到定位时会经历保存地图、加载地图、设置初始姿态等逻辑。

**请求体**：

```json
{
  "mode": "localization"
}
```

**什么时候用**：

+ 你想手动切模式
+ 一般现场更常用 `load_map`，因为它更完整

---

#### `GET /robot/map/list`
**作用**：列出适配层能识别到的地图。  
**封装了什么**：扫描地图目录，判断是否有 `global.pcd + map.yaml + map.pgm` 等文件。  
HumanoidNav 的 `LoadMap` 文档要求地图目录应包含 `global.pcd` 和 `map.yaml`/`map.pgm`。

**什么时候用**：

+ 想知道机器人上有哪些可加载地图
+ 想确认某张图是否 `valid_for_load`

---

#### `GET /robot/map/current`
**作用**：看当前适配层认为“正在使用哪张图”。  
**封装了什么**：读 runtime 里的 `current_map`，没有就尝试从当前地图目录推断。

**注意**：  
它是**适配层当前记录**，不是底层 ROS 一定已经稳定定位好了。

---

#### `POST /robot/map/save`
**作用**：保存当前地图。  
**封装了什么**：底层 `/slam/save_map` service。  
底层 `SaveMap` 主要保存 3D 点云地图 `global.pcd`；2D `map.pgm + map.yaml` 需要配合 `nav2_map_server map_saver_cli`。

**请求体**：

```json
{
  "map_name": "office_a"
}
```

**什么时候用**：

+ 处于建图模式，想把当前建好的图落盘

---

#### `POST /robot/localization/initial_pose`
**作用**：手动重定位。  
**封装了什么**：发布 `/initialpose`。  
`/initialpose` 是标准定位入口，消息类型是 `PoseWithCovarianceStamped`。

**请求体**：

```json
{
  "x": 1.0,
  "y": 2.0,
  "yaw": 1.57,
  "z": 0.0,
  "frame_id": "map"
}
```

**什么时候用**：

+ 地图已经 load，但 `odom_status_code` 还是 `INITIALIZING/FAIL`
+ 机器人实际位置和地图初始位置对不上

---

#### `POST /robot/map/load`
**作用**：按绝对路径加载地图。  
**封装了什么**：底层 `/slam/load_map`，再由适配层把它登记成 `current_map`，并可选等待定位 ready。  
`LoadMap` 的语义不是“只读一张图”，而是：**停止当前模式 → 加载地图 → 设置初始位姿 → 启动定位模式**。

**请求体**：

```json
{
  "map_path": "/opt/fftai/nav/map",
  "x": 0.0,
  "y": 0.0,
  "z": 0.0,
  "yaw": 0.0,
  "wait_for_localization": true
}
```

**什么时候用**：

+ 你知道地图绝对路径
+ 更适合脚本/自动化

---

#### `POST /robot/map/load_by_name`
**作用**：按地图名加载。  
**封装了什么**：

1. 先在适配层地图目录里解析 `map_name`
2. 再调用底层 `/slam/load_map`
3. 更新 `current_map`
4. 可选等待定位 ready

**请求体**：

```json
{
  "map_name": "office_a",
  "x": 0.0,
  "y": 0.0,
  "z": 0.0,
  "yaw": 0.0,
  "wait_for_localization": true
}
```

**什么时候用**：

+ 现场最推荐  
因为不用自己记全路径，只记业务名字

### 4.POI
POI 是业务层概念，不是底层 ROS2 原生对象。  
它本质是“**命名点位**”，底层还是位姿。使用指南里也是按“当前地图 → 保存点位 → 组成 mission”这个流程来写的。

#### `GET /robot/poi/list`
**作用**：列出 POI。  
**支持**：

+ 全部 POI
+ 某张图的 POI
+ 当前地图 POI

**常用**：

```bash
curl "$BASE_URL/robot/poi/list?use_current_map=true"
```

---

#### `GET /robot/poi/{name}`
**作用**：按名字取一个 POI。  
**什么时候用**：

+ 导航前先确认点位是否存在
+ 做业务系统联调时查详情

---

#### `DELETE /robot/poi/{name}`
**作用**：删一个 POI。  
**什么时候用**：

+ 标错点位了
+ 要重新标定

---

#### `POST /robot/poi/save_current`
**作用**：把**当前机器人位姿**保存成一个命名 POI。  
**封装了什么**：从当前 `robot_pose` 读位姿，再写入本地 YAML。  
`robot_pose` 是定位系统发布的 map 坐标系下 6DOF 位姿。

**请求体**：

```json
{
  "name": "front_desk",
  "map_name": "office_a",
  "tags": ["demo"],
  "meta": {
    "note": "front desk"
  }
}
```

**什么时候用**：

+ 机器人已经站在目标位置
+ 想快速标点

---

#### `POST /robot/poi/upsert`
**作用**：手工新增或覆盖 POI。  
**请求体**：

```json
{
  "poi": {
    "name": "charging_station",
    "map_name": "office_a",
    "x": 2.5,
    "y": 4.2,
    "yaw": 1.57,
    "z": 0.0,
    "frame_id": "map",
    "tags": ["charger"],
    "meta": {
      "note": "charging station"
    }
  }
}
```

**什么时候用**：

+ 你已经知道坐标，不需要机器人站过去
+ 做数据导入、批量初始化

---

### 5.导航
#### `POST /robot/navigation/goto_poi`
**作用**：按 POI 名称导航。  
**封装了什么**：

1. 查 POI
2. 取出里面的 `x,y,yaw`
3. 复用 `goto_pose`
4. 创建一个适配层内部导航任务

**请求体**：

```json
{
  "name": "front_desk",
  "map_name": "office_a",
  "force": false
}
```

`force=false` 时会先过 readiness，不 ready 就拦住。  
底层真正执行还是 `navigate_to_pose` action。`NavigateToPose` 的目标是 map 坐标系目标位姿。

---

#### `POST /robot/navigation/goto_pose`
**作用**：直接按坐标导航。  
**封装了什么**：底层 `navigate_to_pose` action。  
`navigate_to_pose` 是 Nav2 的核心动作接口，目标是 `geometry_msgs/PoseStamped`。

**请求体**：

```json
{
  "pose": {
    "x": 1.0,
    "y": 2.0,
    "yaw": 0.78,
    "z": 0.0,
    "frame_id": "map"
  },
  "label": "desk",
  "force": false
}
```

**什么时候用**：

+ 系统直接给你坐标
+ 不想先存成 POI

---

#### `POST /robot/navigation/cancel`
**作用**：取消当前导航。  
**封装了什么**：底层 `/cancel_current_action` service。  
这个 service 的语义就是取消当前正在执行的导航动作。

---

#### `GET /robot/navigation/task`
**作用**：看适配层内部维护的“导航任务对象”。  
**注意**：  
它是适配层自己的任务状态，不是底层 Nav2 的唯一真相。

---

#### `GET /robot/navigation/current_action`
**作用**：看底层当前动作。  
**封装了什么**：

+ `/get_current_action` service 返回
+ `/action_status` topic 快照

`GetCurrentAction` 会给你当前动作名、目标状态码、状态描述；`ActionStatus` 也是对动作状态的持续发布。

**什么时候用**：

+ 任务卡住时看底层到底在干嘛
+ 比 `navigation/task` 更接近底层真实执行状态

---

### 6.Route / Patrol / Mission
这里要特别说一句：

**在这个工程里，mission 和 route 本质上是同一套东西的两种名字。**  
实现上 `missions/upsert` 实际就是往 route 里写，`missions/start` 也是复用 patrol 的顺序执行器。

#### `GET /robot/routes`
**作用**：列 route 列表。  
route 本质是 **POI 名称数组**。

---

#### `POST /robot/routes/upsert`
**作用**：新建/覆盖 route。  
**请求体**：

```json
{
  "route": {
    "name": "morning_patrol",
    "map_name": "office_a",
    "points": ["front_desk", "charging_station"],
    "meta": {
      "note": "morning patrol"
    }
  }
}
```

---

#### `POST /robot/patrol/start`
**作用**：按 route 顺序巡逻。  
**封装了什么**：

1. 检查 readiness
2. 读取 route 里的点位列表
3. 逐个调用 `goto_poi`
4. 支持 `loop=true` 循环执行

**请求体**：

```json
{
  "route_name": "morning_patrol",
  "map_name": "office_a",
  "loop": true,
  "force": false
}
```

---

#### `POST /robot/patrol/stop`
**作用**：停止 patrol。  
**封装了什么**：

+ 给 patrol 线程发 stop
+ 顺便调用 `navigation/cancel`

---

#### `GET /robot/patrol/status`
**作用**：看 patrol 当前跑到哪个点。  
**常见字段**：

+ `route_name`
+ `index`
+ `current_point`
+ `status`

---

#### `GET /robot/missions`
**作用**：列 mission。  
**实际实现**：本质还是 route 列表。

---

#### `GET /robot/missions/context`
**作用**：这个接口非常适合调试。  
会把这些一起返回：

+ 当前地图
+ 当前地图 POI
+ 当前 mission
+ 当前导航任务
+ 当前动作状态
+ localization 状态
+ readiness

使用指南里也明确把它当作“一次性看上下文”的调试接口。

---

#### `POST /robot/missions/upsert`
**作用**：新建/覆盖一个 mission。  
**请求体**和 `routes/upsert` 一样，本质上还是一串 POI 名称。

---

#### `POST /robot/missions/start`
**作用**：启动一个 mission。  
**实际执行逻辑**：顺序跑各个 POI，和 patrol 是同一个执行器。  
使用指南里描述得很清楚：读 mission 点位 → 导到第一个 POI → 成功后继续下一个 → 全部完成后结束。

---

#### `POST /robot/missions/stop`
**作用**：停止当前 mission。  
本质还是停止 patrol 执行器。

---

#### `GET /robot/missions/status`
**作用**：看 mission 当前状态。  
本质上对应当前 patrol/multi-point task 的运行状态。

---

### 7.Aurora 相关
这组接口不是导航 ROS2 栈，而是运动控制/Aurora。

#### `GET /robot/aurora/ping`
**作用**：看 Aurora SDK 有没有装好、能不能连上。  
**什么时候用**：

+ `ensure_stand` 调不通
+ `aurora_connected` readiness 老报错

---

#### `GET /robot/aurora/state`
**作用**：读取 Aurora 当前状态。  
底层 AuroraClient 可以读 FSM 等状态；例如 `get_fsm_state()` 能返回当前全身 FSM。文档里列了常见状态：`1=关节站立`，`2=PD站立`，`3=RL运动` 等。

---

#### `POST /robot/aurora/fsm`
**作用**：直接设置全身 FSM。  
**封装了什么**：AuroraClient `set_fsm_state(state)`。  
Aurora 文档给出的有效值包括：

+ `1` 关节站立
+ `2` PD 站立
+ `3` RL 运动
+ `9` 安全保护
+ `10` 用户指令状态

**请求体**：

```json
{
  "fsm_state": 2
}
```

---

#### `POST /robot/aurora/ensure_stand`
**作用**：让机器人进入站立态。  
**封装了什么**：业务层 AuroraService 的“确保站立”逻辑。  
现场通常把它当成导航前的准备动作先调用一次。

---

#### `POST /robot/aurora/stop_motion`
**作用**：立即停运动。  
**封装了什么**：Aurora 速度清零/停止动作。

---

### 8.这些接口该按什么顺序用
你现场最常见的顺序，建议就按这个来：

#### 场景 A：单点导航
1. `GET /healthz`
2. `POST /robot/aurora/ensure_stand`
3. `GET /robot/readiness`
4. `GET /robot/map/list`
5. `POST /robot/map/load_by_name`
6. 必要时 `POST /robot/localization/initial_pose`
7. `GET /robot/localization/status`
8. `POST /robot/navigation/goto_poi` 或 `POST /robot/navigation/goto_pose`
9. `GET /robot/navigation/task`
10. `GET /robot/navigation/current_action`
11. 必要时 `POST /robot/navigation/cancel`

#### 场景 B：先标点，再跑 mission
1. `POST /robot/map/load_by_name`
2. `POST /robot/poi/save_current` 保存多个点
3. `POST /robot/missions/upsert`
4. `POST /robot/missions/start`
5. `GET /robot/missions/status`
6. `GET /robot/missions/context`
7. `POST /robot/missions/stop`
