# 一、先用一句话概括这套系统
这套 HumanoidNav 本质上是一个基于 **ROS2 Humble** 的完整导航系统，核心由 5 层组成：

1. **传感器输入层**：IMU、激光雷达、点云、扫描
2. **建图/定位层**：SLAM / Localization（lightning / unified）
3. **地图层**：3D 点云地图 + 2D 占用栅格地图
4. **导航控制层**：Nav2（规划、控制、行为树、速度输出）
5. **监控与业务封装层**：health / events / monitor / raw_client / HTTP adapter

官方文档里对系统的总体分层写得很清楚：  
传感器输入（`/imu`、`/lidar_points`）进入 3D 激光建图/定位，再生成 `/map` 和 `/optimize_map`，之后由 Nav2 做全局规划、局部控制和行为树导航，最后输出速度命令。

---

# 二、ROS2 里这套系统的基本组织方式
先把 ROS2 这几个最重要的概念理顺，不然后面很容易乱。

## 1. Node
**Node 就是一个功能进程/组件。**

比如你现场看到的这些，都是 node 或由 launch 拉起的组件：

+ `rslidar_sdk_node`
+ `pointcloud_to_laserscan_node`
+ `health_node`
+ `events_node`
+ `run_unified_online`
+ `map_server`
+ `planner_server`
+ `controller_server`
+ `bt_navigator`
+ `monitor_node`
+ `raw_client_node`

每个 node 各做一件事，然后通过 ROS2 的 topic / service / action 联通。

---

## 2. Topic
**Topic 是持续流动的数据流。**

适合“不断发布状态/传感器数据”的场景。  
这套系统里最重要的 topic，大致分几类：

### 传感器类
+ `/imu`
+ `/rslidar_points`
+ `/scan`
+ `/cloud_registered_gravity`
+ `/segmented_groundless_points`

### 定位/地图类
+ `/slam/mode_status`
+ `/robot_pose`
+ `/odom`
+ `/odom_status_code`
+ `/odom_status_score`
+ `/map`
+ `/optimize_map`

### 导航/控制类
+ `/plan`
+ `/cmd_vel`
+ 以及内部的平滑速度输出链

### 监控类
+ `/Humanoid_nav/health`
+ `/Humanoid_nav/events`

---

## 3. Service
**Service 是一次请求、一次响应。**

适合“切模式、加载地图、查询状态”这种命令式操作。

这套系统最关键的 service 有：

+ `/slam/set_mode`
+ `/slam/load_map`
+ `/slam/save_map`
+ `/get_current_action`
+ `/cancel_current_action`

这里你前面已经踩过两个典型坑：

+ service 类型应该是 `fourier_msgs/...`，不是 `humanoid_nav_msgs/...`
+ `LoadMap` 的参数字段是 `map_path`，不是 `path`。

---

## 4. Action
**Action 是“长时间执行、能反馈进度的任务”。**

这套系统里最核心的 action 是：

+ `/navigate_to_pose`

它不是一次性返回，而是：

+ 先接收目标点
+ 再持续返回反馈
+ 最后返回成功/失败结果。

所以导航目标本质上不是 service，而是 action。

---

## 5. Namespace
**Namespace 是“给一台机器人整套 ROS 接口统一加前缀”。**

文档里写得很清楚：  
如果算法包运行在自定义 namespace 下，那么所有 node/topic/service/action 都会统一增加前缀，但消息体结构不变。

你现在这台机器用的 namespace 是：

```plain
/GR301AA0025
```

所以：

+ `/slam/set_mode` → `/GR301AA0025/slam/set_mode`
+ `/robot_pose` → `/GR301AA0025/robot_pose`
+ `/navigate_to_pose` → `/GR301AA0025/navigate_to_pose`

这也是为什么你现场所有命令都要带这个前缀。

# 四、TF 到底是什么，为什么你这里老出问题
你现在最需要把 TF 理解透。

## 1. TF 是什么
TF 本质上是**坐标系关系树**。  
导航系统不是只关心“有没有点云”，而是更关心：

+ 这些点云在什么坐标系下
+ 机器人身体在哪
+ 雷达安装在身体哪个位置
+ 地图坐标和本体坐标怎么对应

这些都靠 TF 描述。

---

## 2. 这套导航里最关键的 frame
理论上最常见的是：

+ `map`
+ `odom`
+ `base_link`
+ `lidar_link`

### 它们的意义
+ `map`：全局地图坐标系
+ `odom`：里程计坐标系
+ `base_link`：机器人机体中心/底盘坐标系
+ `lidar_link`：雷达安装坐标系

---

## 3. 常见关系
通常应该是：

```plain
map -> odom -> base_link -> lidar_link
```

### 各段的典型来源
+ `map -> odom`：定位系统给
+ `odom -> base_link`：里程计/机体状态给
+ `base_link -> lidar_link`：静态 TF / 机器人模型给

---

## 4. 你现在的问题为什么卡在这里
你已经实测到：

+ `odom -> base_link` 不存在
+ `base_link -> lidar_link` 不存在
+ `/odom` 没有样本
+ `/tf` 和 `/tf_static` 虽然有 publisher，但你需要的 frame 没建立起来。

所以这套系统当前更像是：

+ 导航大脑（Nav2、SLAM）已经起来
+ 但“机器人身体”的状态坐标链没接进来
+ 导致定位初始化失败
+ 进而 `robot_pose / odom_status_code` 都不给数据

# 六、你当前这套系统的几个“关键认识”
## 1. `run_real.sh` 已经是总入口
它不是一个小脚本，而是一个“整套 bringup wrapper”。  
从你机器上实际进程就能看出，它已经把 navigation、localization、health、events、perception、Nav2 整体拉起来了。

所以以后别再把它和单独起 `humanoidnav.navigation.launch.py`、`lightning unified.launch.py` 混着理解。

---

## 2. `raw_client_node` 不是导航核心
它只是桥接层。  
它订阅底层很多状态，再暴露控制接口。  
所以：

+ 它能看到 `/navigate_to_pose`
+ 不代表定位一定正常
+ 它本身健康，也不代表 SLAM ready

---

## 3. `GetCurrentAction` 成功，只能说明“控制平面通了”
不能说明“定位平面通了”。

你前面已经看到：

+ `GetCurrentAction` 能返回 `Idle`
+ 但 `robot_pose / odom_status_code / slam_mode_status` 还是没有数据。

所以：

+ service/action 层通了
+ ≠ localization 就 ready 了

---

## 4. health 这条现在有明确异常
`/Humanoid_nav/health` 正常应该是 `fourier_msgs/msg/HealthInfo`，但你现场同时看到它还有 `sensor_msgs/msg/Imu`，而且 `raw_client_node` 就把它订成了 `Imu`。这是一个很明确的桥接/配置错误。

这个错误会干扰 health 相关判断，但它不是你当前“TF 不通”的唯一原因。

# 七、你现在应该怎么理解“问题到底在哪一层”
如果按层看，你现在的问题是这样的：

## 没问题的层
+ 地图文件层：基本没问题
+ service/action 类型层：已基本理顺
+ navigation bringup 层：已经起来

## 有问题的层

也就是说：

现在不是“整个导航 SDK 不工作”，而是“导航 SDK 已经起来，但机器人本体状态没有正确喂进来”。
